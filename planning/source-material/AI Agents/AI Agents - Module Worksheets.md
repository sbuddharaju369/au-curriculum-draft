# Appendix — Numerical Drills, Difficulty Index, and Self-Assessment Rubrics

The worksheets above test comprehension. This appendix tests *fluency with numbers* — the back-of-envelope arithmetic that separates "I read the slides" from "I can architect an agent." Six drills (D0–D5), one per module, each ~10 minutes with worked solutions. Plus a difficulty index for every existing worksheet question, and self-assessment rubrics for grading your own answers.

## How to use this appendix

1. Complete each module's worksheet first.
2. Attempt the matching drill *without* opening a calculator — commit to a number before checking.
3. Score yourself against the rubric.
4. If you can solve any 4 of 6 drills within 12 minutes each, you've mastered the quantitative side of the course.

## D0 — Agent-call multiplier (Module 0)

A chatbot session averages **5 model calls** of 600 tokens each. An agent session for the same user task averages **12 model calls** of 800 tokens each. Token price (output): **$3 / M tokens**.

(a) Per-session cost for each? (b) Multiplier? (c) At 1M DAU × 3 sessions/day, annual cost difference?

**Solution.**

(a) Chatbot: 5 × 600 × $3/M = **$0.009/session**. Agent: 12 × 800 × $3/M = **$0.0288/session**.

(b) **~3.2× cost multiplier per session**.

(c) Difference per day: 1M × 3 × ($0.0288 − $0.009) = $59,400/day. Annual: **~$21.7M extra**. This is the cost of the loop, paid for entirely by inference.

*Concept ID: `agent-loop`, `inference-cost`.*

## D1 — Model-routing arithmetic (Module 1)

An agent task does **20 model calls × 800 tokens** average. Naive design routes everything to frontier (Claude Opus, $15/M output). Smart routing: 3 calls to Opus ($15/M), 12 to Haiku ($1/M), 5 to open Llama 70B ($0.30/M).

(a) Naive per-task cost? (b) Smart-routing per-task cost? (c) Savings at 100K tasks/day, annualized?

**Solution.**

(a) 20 × 800 × $15/M = **$0.24/task**.

(b) (3 × 800 × $15 + 12 × 800 × $1 + 5 × 800 × $0.30) / 1,000,000 = (36,000 + 9,600 + 1,200) / 1M = **$0.0468/task**.

(c) Savings per task: $0.193. Daily: $19,300. Annual: **~$7M/year**, just from routing.

*Concept ID: `agent-loop`, `inference-cost`.*

## D2 — Tool-call compounding (Module 2)

An agent uses **8 tool calls** per task. Each tool call independently has success rate **p**.

(a) End-to-end success rate for p = 0.90, 0.95, 0.99? (b) If your product target is 90% end-to-end success, what minimum per-call rate must you achieve? (c) Why is investing in tool reliability higher-leverage than investing in model intelligence?

**Solution.**

(a) p=0.90: 0.90⁸ ≈ **43%**. p=0.95: 0.95⁸ ≈ **66%**. p=0.99: 0.99⁸ ≈ **92%**.

(b) Need 0.9^(1/8) ≈ **0.9869 → 98.7%**.

(c) Tool reliability compounds *multiplicatively* across calls; model improvements compound *linearly*. A 4-point gain in per-call rate (95→99) turns a 66% product into a 92% product — a 30-point end-to-end gain.

*Concept ID: `tool-use`, `mcp`.*

## D3 — Sandbox blast radius (Module 3)

An agent runs shell commands. Three deployment modes:

- **Mode A**: runs on the host with user privileges (no sandbox).
- **Mode B**: Docker container, host volumes mounted (partial sandbox).
- **Mode C**: ephemeral container, no host mounts, scoped network egress (full sandbox).

A successful indirect-prompt-injection occurs once every **10,000 invocations**. Expected cost of a single uncontrolled compromise: **$50,000** (incident response + reputational + customer notification).

(a) Annual expected loss in each mode for 1M invocations/year (assume B reduces consequence cost by 80%, C by 99%)? (b) Cost-benefit if Mode C adds $5/1000 invocations of overhead?

**Solution.**

(a) Incidents/year: 1M / 10,000 = **100 incidents/year**. Mode A: 100 × $50K = **$5M/year**. Mode B: 100 × $10K = **$1M/year**. Mode C: 100 × $500 = **$50K/year**.

(b) Mode C overhead: 1M × $0.005 = **$5K/year**. Net savings vs. Mode A: $5M − $50K − $5K ≈ **$4.95M/year**. Sandboxing pays for itself 1000×.

*Concept ID: `sandboxing`, `prompt-injection`, `least-privilege`.*

## D4 — Orchestration speedup (Module 4)

A 5-agent workflow: planner → {researcher, coder} → tester → reviewer. Each agent: 8 model calls × 1 sec/call.

(a) Serial wall-clock time? (b) With parallelism (researcher + coder concurrent after planner)? (c) With caching (skip unchanged researcher work on retries — 30% of runs are retries)? (d) Average wall-clock across 100 runs?

**Solution.**

(a) Serial: 5 × 8 = **40 sec**.

(b) Parallel: 8 (planner) + max(8, 8) (research/code parallel) + 8 (test) + 8 (review) = **32 sec**.

(c) On a retry, skip researcher: 8 + 8 (coder only) + 8 + 8 = **24 sec**.

(d) 70 × 32 + 30 × 24 = 2240 + 720 = 2960; avg = **29.6 sec**. That's a 1.35× win on average, ~1.7× on cached retries.

*Concept ID: `orchestration`, `multi-agent-systems`.*

## D5 — Pricing-model break-even (Module 5)

You charge **$50/user/month**. Cost per agent task: **$0.04**. User distribution: 50% light (10 tasks/month), 30% medium (200 tasks/month), 20% heavy (2,000 tasks/month).

(a) Cost per user in each tier? (b) Per-tier margin? (c) Weighted average margin per user? (d) What happens to (c) if heavy users grow to 40% of base?

**Solution.**

(a) Light: 10 × $0.04 = **$0.40**. Medium: 200 × $0.04 = **$8**. Heavy: 2000 × $0.04 = **$80**.

(b) Light: +$49.60. Medium: +$42. Heavy: **−$30 (loss!)**.

(c) 0.5 × 49.60 + 0.3 × 42 + 0.2 × (−30) = 24.80 + 12.60 − 6 = **+$31.40/user/month**.

(d) New mix: 40% light, 20% medium, 40% heavy. 0.4 × 49.60 + 0.2 × 42 + 0.4 × (−30) = 19.84 + 8.40 − 12 = **+$16.24/user/month**. Margin nearly halved. Per-seat pricing breaks down as heavy users scale. **Forces consumption-based or usage-capped pricing.**

*Concept ID: `agent-economics`, `inference-cost`.*
